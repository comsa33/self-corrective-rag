\paragraph{Complex multi-hop questions.}
\ours{} achieves the best performance on 2WikiMultiHopQA (F1 0.554, $+0.069$ over Loop) and MuSiQue (F1 0.453, $+0.051$ over Loop).
These improvements are consistent across all three metrics (EM, F1, Judge), indicating genuine quality gains rather than metric-specific artifacts.
The LLM-as-Judge scores show even larger gaps: $+0.080$ on 2Wiki and $+0.040$ on MuSiQue, suggesting that the agentic approach produces answers that are more semantically correct even when token overlap is moderate.

\paragraph{Simple 2-hop questions.}
On HotpotQA, all pipeline variants perform within a narrow range (F1 0.605--0.726), with Single-Pass slightly leading.
This suggests that 2-hop questions do not require iterative refinement---a single well-preprocessed retrieval pass is sufficient, and the additional overhead of agentic reasoning provides no benefit.

\paragraph{Enterprise documents.}
On FinanceBench, Loop Refinement marginally outperforms \ours{} in F1 (0.414 vs.\ 0.412, $\Delta = -0.002$), while \ours{} leads in Judge score (0.900 vs.\ 0.840).
The near-identical F1 scores suggest that both refinement approaches extract similar information, but the agentic approach may produce more naturally phrased answers that the LLM judge favors.

\paragraph{CRAG Replica underperformance.}
The CRAG Replica consistently ranks lowest, which is expected: our implementation replaces CRAG's web search fallback with internal re-retrieval (to match our enterprise-constrained setting) and uses LLM-based evaluation instead of a fine-tuned T5-large evaluator.
These architectural differences explain the gap with reported CRAG results.

\paragraph{Key finding.}
The agentic advantage is \emph{complexity-dependent}: improvements grow with question complexity (number of reasoning hops), while simple questions show no benefit.
This pattern is consistent with the hypothesis that autonomous tool selection provides value primarily when the retrieval task is non-trivial and benefits from adaptive strategy.

\ours{} requires 14--17 seconds per question, approximately 4--5$\times$ slower than Loop Refinement and 30--50$\times$ slower than Naive RAG.
This overhead stems from the ReAct reasoning loop: each iteration involves LLM inference for thought generation, tool selection, and observation processing.
With an average of 5.5--5.8 tool calls per question (see \S\ref{subsec:rq2}), the agent performs substantially more LLM calls than the fixed-loop approach.

We discuss the latency--quality trade-off and practical implications in \S\ref{sec:discussion}.

\paragraph{Search dominance.}
The agent calls \texttt{search\_passages} 3.5--3.9 times per question, making it the most frequently used tool.

% ---------------------------------------------------------------------------
\subsubsection{Trajectory Patterns}
\label{subsubsec:trajectories}
% ---------------------------------------------------------------------------

Analysis of ReAct trajectories reveals three dominant behavioral patterns:

\begin{enumerate}
    \item \textbf{Decompose--Search--Evaluate} (most common): The agent decomposes the question into sub-questions, searches each independently, and evaluates the combined passage set. This is the canonical multi-hop retrieval strategy.

    \item \textbf{Search--Evaluate} (simple cases): For questions identified as single-hop, the agent skips decomposition and directly searches with the preprocessed query before evaluating.

    \item \textbf{Decompose--Search--Evaluate--Search--Evaluate} (refinement loop): When the initial evaluation score falls below the quality threshold, the agent performs an additional search round using the evaluation feedback (suggested query, keywords) and re-evaluates.
\end{enumerate}

Notably, the agent \emph{spontaneously develops} these structured protocols through ReAct reasoning, despite having access to a broader set of tools and no hard-coded execution sequence.
The protocol is suggested in the agent's signature docstring but not enforced---the agent could deviate but consistently converges on these patterns.

\paragraph{Domain-specific tool use.}
On FinanceBench, the agent occasionally invokes \texttt{calculate} (0.1 calls/question) for numerical computations such as margin calculations and year-over-year comparisons.
This tool is virtually unused on the Wikipedia-based datasets, demonstrating that the agent adapts its tool selection to the domain.

\paragraph{Structural benefit: Manual $\to$ DSPy Unopt.}
The most striking finding is that simply replacing hand-crafted JSON prompt templates with DSPy typed Signatures yields substantial improvements \emph{without any optimization}: F1 increases of $+0.099$ (HotpotQA), $+0.054$ (2Wiki), and $+0.207$ (FinanceBench).
The FinanceBench improvement is particularly dramatic, from 0.197 to 0.404 (more than doubling).
This suggests that DSPy's structured prompting---with typed input/output fields, descriptive field annotations, and task-specific docstrings---provides a more effective interface for LLM reasoning than manually constructed prompts.

\paragraph{Optimization benefit: Unopt $\to$ Bootstrap/MIPROv2.}
Automatic optimization provides additional gains, but the effect is \emph{dataset-dependent}:

\begin{itemize}
    \item \textbf{BootstrapFewShot} excels on structured multi-hop tasks: best F1 on 2Wiki (0.632, $+0.083$ over Unopt) and FinanceBench (0.424, $+0.020$). Successful execution traces as few-shot examples help the model recognize multi-hop retrieval patterns.

    \item \textbf{MIPROv2} excels on complex reasoning: best F1 on MuSiQue (0.577, $+0.110$ over Unopt), the hardest dataset. Optimized instructions apparently help with deeper reasoning chains.

    \item On HotpotQA, Unopt already achieves strong performance (0.717), and optimization provides marginal or no improvement---a ceiling effect.

    \item \textbf{Over-optimization risk}: On FinanceBench, MIPROv2 (0.368) \emph{underperforms} Unopt (0.404), suggesting that domain-specific data may benefit from conservative optimization or domain-adapted training examples.
\end{itemize}

\paragraph{Latency reduction.}
An unexpected benefit of optimization is reduced inference latency: Bootstrap and MIPROv2 variants complete in 2--3 seconds per question versus 19--24 seconds for Manual/Unopt (Table~\ref{tab:rq5_latency}).
Pre-computed few-shot demonstrations reduce the amount of in-context reasoning required, effectively trading optimization-time compute for inference-time efficiency.

\paragraph{Key finding.}
DSPy provides a two-level contribution: (1) the \emph{structural benefit} of typed Signatures alone is the single largest improvement in our experiments ($+0.1$--$0.2$ F1), and (2) automatic optimization provides additional dataset-dependent gains.
This supports the practical value of declarative LM programming for RAG systems.
